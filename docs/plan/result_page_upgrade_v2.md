# Revised Implementation Plan: Result Page Social Integration

This document outlines a simplified, production-safe implementation for the Result Page social upgrade, aligning with the actual Firestore data flow identified in the codebase.

## 1. Analysis of Current Data Flow

- **Store**: `OnboardingContext.tsx` acts as the single source of truth for the participant's state during the sesssion.
- **Sync**: `useParticipant.ts` maintains a real-time Firestore `onSnapshot` listener on the current participant's document (`rooms/{roomId}/participants/{participantId}`).
- **Consumption**: `ResultStep.tsx` pulls data directly from the context hook.

### Inferred Firestore Participant Structure
Based on `useParticipant.ts`, the document contains:
```typescript
{
  username: string,
  pronoun: string,
  mood: string,
  status: "generating_questions" | "answering" | "waiting_for_ai" | "processing_ai" | "ready" | "error",
  avatarUrl?: string,      // Generated by backend AI
  aiTopics?: string[],     // Generated by backend AI
  matched_participants?: [ // New field for integration
    {
      uid: string,
      username?: string,
      avatarUrl?: string
    }
  ]
}
```

## 2. Simplification Strategy

> [!NOTE]
> We are removing any logic that attempts to fetch external participant documents. All data required for the Result Page is assumed to be pushed into the `matched_participants` field by the matching engine.

- **REMOVED**: UID-to-Participant fetch mapping.
- **REMOVED**: Dependencies on external collections outside the current room.
- **REMOVED**: Complex async resolution in the UI layer.

## 3. Proposed Changes

### State Management
#### [MODIFY] [OnboardingContext.tsx](file:///d:/Github_Repos/IceBreaker/src/context/OnboardingContext.tsx)
- Add `matchedParticipants` state:
  ```typescript
  matchedParticipants: Array<{ uid: string; username?: string; avatarUrl?: string }>;
  ```

### Data Sync
#### [MODIFY] [useParticipant.ts](file:///d:/Github_Repos/IceBreaker/src/hooks/useParticipant.ts)
- Add `setMatchedParticipants` to the hook dependencies.
- Update `onSnapshot` to sync `data.matched_participants` if it exists.

### UI Update
#### [MODIFY] [ResultStep.tsx](file:///d:/Github_Repos/IceBreaker/src/components/mobile/ResultStep.tsx)
- **Remove**: The entire `Button` component used for "Add to Wallet".
- **Reorder**: Shift the match section (`{/* New Match Section */}`) to appear directly after the Card Header and before the "Info & Topics" section.
- **Dynamic Render**:
  - Map over `matchedParticipants`.
  - Use `avatarUrl` if present, fallback to emoji symbol (`🦊`).
  - Use `username` if present, fallback to `t('anonymous')` or appropriate placeholder.
- **Fallback State**: If `matchedParticipants` is empty or undefined, render the static 🐶🐱🐰 demos as a "community placeholder" to ensure the UI stays active.

## 4. Edge Case Handling

- **Missing Data**: No `matched_participants` field in Firestore → Shows default 🐶🐱🐰 demo.
- **Empty List**: `matched_participants: []` → Shows default 🐶🐱🐰 demo.
- **Partial Metadata**: Match has UID but no `avatarUrl` → Shows 🦊 emoji.

# 🗺️ Start Session Frontend Plan (Simplified)

## 1. 🧠 DESIGN PRINCIPLE
The frontend acts purely as a presentation layer. Backend greedy matching runs continuously from the moment users join a room. The frontend "Start Session" action does not trigger matching; it simply acts as a UI visibility gate to reveal the matches that have already been generated.

---

## 2. 🔘 START SESSION BUTTON (Lobby / Room Page)

### Expected Behavior:
- Host clicks "Start Session".
- UI immediately changes state to reveal the matches.
- No artificial delays, transition screens, or loading states are used.

### Clean Component Implementation

```tsx
// src/components/hall/RoomPage.tsx
import React, { useState } from 'react';
import { Button } from '@/components/ui/Button';
import { useTranslation } from '@/context/LanguageContext';
import { LobbyView } from './LobbyView';
import { ResultPage } from './ResultPage';

export type SessionState = 'waiting' | 'matched' | 'closed';

export function RoomPage() {
  const { t } = useTranslation();
  const [sessionState, setSessionState] = useState<SessionState>('waiting');

  const handleStartSession = () => {
    // Instantly reveal matches generated by the backend
    setSessionState('matched');
  };

  // Render Sub-view
  if (sessionState === 'waiting') {
    return (
      <LobbyView>
        <Button
          onClick={handleStartSession}
          className="w-full mt-6"
        >
          {t('startSession')}
        </Button>
      </LobbyView>
    );
  }

  if (sessionState === 'matched') {
    return <ResultPage />;
  }

  return <EndScreen />; // Fallback for 'closed'
}
```

## 3. 🔌 FUTURE BACKEND INTEGRATION (IMPORTANT)

When the backend architecture stabilizes and Firestore synchronization is required, the component will use a real-time `onSnapshot` listener to sync the global state instead of relying purely on the local `useState`.

### The Integration Contract Hook

```tsx
// Future integration snippet (in a custom hook e.g., useRoomSession.ts)
import { useEffect } from 'react';
import { doc, onSnapshot } from 'firebase/firestore';
import { db } from '@/lib/firebase';

export function useRoomSession(roomId: string) {
  const [sessionState, setSessionState] = useState<SessionState>('waiting');

  useEffect(() => {
    if (!roomId) return;
    
    const roomRef = doc(db, 'rooms', roomId);
    const unsubscribe = onSnapshot(roomRef, (docSnap) => {
      if (docSnap.exists()) {
        const rawStatus = docSnap.data().status;
        
        // Expected Mapping Translation
        const statusMap: Record<string, SessionState> = {
          'waiting': 'waiting',
          'matched': 'matched',
          'closed': 'closed'
        };
        
        setSessionState(statusMap[rawStatus] || 'waiting');
      }
    });

    return () => unsubscribe();
  }, [roomId]);

  return { sessionState, setSessionState };
}
```

**Note:** The Start Session button click will then call `updateDoc(roomRef, { status: 'matched' })` to trigger the update across all participant screens.

---

## 4. ⚠️ DESIGN CONSTRAINTS

1. **Instant Feedback:** The UI guarantees immediate feedback using `setSessionState('matched')` directly in the `onClick` handler.
2. **Backend Authority:** The frontend assumes the backend is already assembling matches. The UI's only job is to unhide the result page.

## 5. 🎉 RESULT PAGE DESIGN

The Result Page is a public reveal moment displayed on the Hall screen. Its goal is to immediately draw participants' attention to their mobile phones to see their dedicated match. There are **no loading states or matching timers**—this page appears instantaneously.

# Specification: Backend Integration Contract (Ice-Breaker AI Pipeline)

## 1. Overview
This document defines the interface and lifecycle for the Ice-Breaker onboarding and AI generation pipeline. The system is **Firestore-driven**, using the `status` field as the primary synchronization mechanism between the React frontend and the Cloud Run backend worker.

---

## 2. Participant Data Structure
**Location**: `rooms/{roomId}/participants/{participantId}`

### 2.1 Backend-Oriented Schema
| Field | Type | Owner | Description |
| :--- | :--- | :--- | :--- |
| **`username`** | `string` | Frontend | Initial profile data. |
| **`mood`** | `string` | Frontend | Initial profile data (emoji/label or "Photo Taken"). |
| **`pronoun`** | `string` | Frontend | Initial profile data. |
| **`status`** | `string` | Both | State machine controller (see section 3). |
| **`questions`** | `Array<{ id: string, text: string }>` | **Backend** | Personalized questions generated by Gemini. |
| **`qa`** | `Array<{ questionId: string, question: string, answer: string }>` | Frontend | User answers paired with full question context. |
| **`aiTopics`** | `string[]` | **Backend** | Final ice-breaking topics for the name card. |
| **`avatarUrl`** | `string` | **Backend** | URL to the generated personalized animal avatar. |
| **`errorMessage`**| `string` | **Backend** | Field for relaying failure context to the UI. |

---

## 3. Status State Machine (Lifecycle)

| Status | Trigger (Written by) | Backend Action |
| :--- | :--- | :--- |
| **`generating_questions`**| Frontend (Initial Write) | Read profile -> Generate 2-3 personalized questions -> Write `questions` array. |
| **`answering`** | **Backend** (After questions write) | None. Waiting for user input in frontend. |
| **`waiting_for_ai`** | Frontend (After QA submit) | Read `qa` entries -> Generate `aiTopics` & `avatarUrl` -> Write final results. |
| **`ready`** | **Backend** (Success) | None. Frontend displays the name card. |
| **`error`** | **Backend** (Failure) | None. Frontend displays `errorMessage`. |

---

## 4. End-to-End Flow Detail

### Step 1: Initial Profile (Trigger: `generating_questions`)
- **Frontend** creates the participant doc with profile fields.
- **Backend Worker** detects `status == "generating_questions"`.
- **Constraint**: Backend MUST generate unique `id` for each question.

### Step 2: Question Generation (Transition: `answering`)
- **Backend** writes the `questions` array.
- **Backend** sets `status` to `answering`.
- **Frontend** automatically renders the question UI.

### Step 3: Answer Submission (Trigger: `waiting_for_ai`)
- **Frontend** constructs a structured `qa` array.
- **Requirement**: `qa` entries MUST include the full `question` text. Gemini requires this context to understand the answers.
- **Frontend** sets `status` to `waiting_for_ai`.

### Step 4: Final AI Output (Transition: `ready`)
- **Backend Worker** detects `status == "waiting_for_ai"`.
- **Backend** processes `qa` context.
- **Backend** writes `aiTopics` and `avatarUrl`.
- **Backend** sets `status` to `ready`.

---

## 5. Critical Integration Rules
1. **Per-User Flow**: All generation happens at the `participant` level. Do not write questions to the `room` document.
2. **Atomic Updates**: Use `updateDoc` (or equivalent) to avoid overwriting timestamp fields (`createdAt`, `expiresAt`) managed by other processes.
3. **ID Mapping**: The `questionId` in the `qa` array MUST exactly match the `id` provided in the `questions` list.
4. **Resilience**: The backend worker must handle re-polling gracefully. If it finds a document in a processing state for too long, it should either retry or set `status: "error"`.
